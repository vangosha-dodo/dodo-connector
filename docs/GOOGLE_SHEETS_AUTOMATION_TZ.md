# ТЗ: функции для Google Sheets automation

Дата анализа: 2026-06-17.

Этот документ фиксирует, какие показатели нужны в трех пользовательских
таблицах, откуда их можно брать и какие функции нужно добавить в Bridge/worker.
Сами Google Sheets сейчас не заполняем: сначала только read-only discovery,
dry-run и проверка диффа.

## Общая граница безопасности

- Dodo IS и Superset остаются read-only источниками: читаем отчеты, страницы и
  API, но ничего не меняем в Dodo IS.
- ChatGPT Action остается read-only. Записи в Google Sheets не должны быть
  доступны агенту ChatGPT напрямую.
- Запись в Google Sheets разрешается только внутреннему scheduler/worker после
  отдельного включения job и только в allowlist диапазоны.
- Каждая job обязана иметь `dry_run=true`: показать строки, формулы и диапазоны,
  которые будут изменены, без записи.
- Для каждой записи нужен audit log: `job_name`, `run_id`, дата отчета,
  spreadsheet id, sheet, range, row count, source snapshot hash, результат.
- Нужны idempotency keys и блокировка job, чтобы повторный запуск не создавал
  дубли.
- Таблицы 1 и 3 имеют timezone `America/Los_Angeles`, но бизнес-расписание
  задано по Москве. В расчетах дат использовать `Europe/Moscow`, не timezone
  файла.

## Общие технические компоненты

1. `automation_worker`
   - расписание jobs;
   - ручной запуск job;
   - job locks;
   - audit logs;
   - режим dry-run.

2. `google_sheets_client`
   - сервисный аккаунт или OAuth;
   - allowlist spreadsheet/sheet/range;
   - чтение заголовков, формул и текущих строк;
   - append/upsert/update с проверкой диапазона.

3. `dodo_office_manager_client`
   - web-сессия Office Manager;
   - переавторизация кодом из почты по уже реализованному сценарию;
   - извлечение отчетов/страниц без записи;
   - нормализация HTML/CSV/XLSX/JSON ответа в структурированные записи.

4. `sheet_diff_service`
   - сравнить текущие строки и новые данные;
   - отдать preview: что добавится, что обновится, что будет пропущено;
   - подсветить потенциальные конфликты с ручными колонками.

## Функция 1. Ежедневная выгрузка зарплаты курьеров

### Цель

Ежедневно в 00:30 Мск за предыдущий отчетный день выгружать из Dodo IS Office
Manager отчет по зарплате курьеров и дописывать его в таблицу
`А-2 Зп курьеров (с 30.03.26)_агент`, лист `Ежедневная выгрузка`.

### Источник

- Основной источник: Dodo IS Office Manager.
- Путь: `Отчеты -> Заработная плата`.
- Фильтры:
  - дата: один отчетный день;
  - пиццерия: каждая разрешенная пиццерия;
  - тип: `Курьер`.
- Dodo API может дать часть данных по сменам/доставкам, но для этой таблицы
  главный источник именно Office Manager report, потому что нужны ставки,
  начисления и тип оформления.

### Целевой лист и показатели

Лист `Ежедневная выгрузка`, диапазон структуры `A:BF`, 58 колонок:

- исходные поля отчета Dodo IS: `Дата начала смены`, `Время начала смены`,
  `Время окончания смены`, `Дата окончания смены`, `Номер телефона`,
  `Фамилия`, `Имя`, `Отчество`, `ИНН`, `Табельный номер`, `Категория`,
  `Ставка в час`, `Ставка за заказ`, `Стаж, мес.`,
  `Коэф-т особого дня`, `Ночной коэф-т`, `Ночные часы`, `Дневные часы`,
  `Праздничные часы`, `Количество заказов`, `ЗП, ночь`, `ЗП, день`,
  `ЗП, за заказ`, `ЗП, премия за стаж`, `Итого`, `Id сотрудника`,
  `UUId сотрудника`, `Тип оформления`, `Пиццерия`, `Расстояние (км)`,
  `ЗП, километраж`, `Поездки (количество)`, `ЗП, поездки`;
- ручные или полуавтоматические поля: `ЗП, премия`,
  `Комментрий к премиям`, `Невыход по графику / не нашел замену`,
  `Опоздание`, `Кол-во проблем с заказами`,
  `Вид доплаты за праздники / пики`,
  `Кол-во заказов праздничных / пиковых`,
  `Кол-во праздничных / пиковых часов`,
  `Коэффициент праздничных / пиковых часов/заказов`,
  `Кол-во довозов (смена адреса)`, `Ставка за ГСМ, руб/км`,
  `Погода (Обычно / Сложно / Экстрим)`, `Вид доплаты за погоду`,
  `Премия за разгрузку`;
- формульные поля: `Сумма доплаты праздничных / пиковых часов/заказов`,
  `Премия за довозы`, `Компенсация ГСМ`,
  `Компенсация ТО (5 руб./км) только СМЗ и ИП`,
  `Надбавка за заказ`, `Надбавка за погоду км`,
  `Надбавка за погоду ч`, `Итого зп`, `Неделя`, `Месяц`, `Год`.

### Правила расчетов, обнаруженные в листе

- `Премия за довозы = Кол-во довозов * Ставка за заказ`.
- `Компенсация ГСМ = Расстояние (км) * Ставка за ГСМ, руб/км`.
- `Компенсация ТО = 5 * Расстояние (км)` только для `Самозанятый` и `ГПХ`
  с заполненным ИНН.
- Надбавка за погоду:
  - если вид `Заказ`, то `Сложно = 50% ставки за заказ`, `Экстрим = 100%`;
  - если вид `Км и час`, то `Сложно = 20% ГСМ + 50% часовой ставки`,
    `Экстрим = 50% ГСМ + 100% часовой ставки`.
- `Итого зп = ЗП день + ЗП за заказ + ЗП премия + премия за довозы +
  компенсация ГСМ + компенсация ТО + надбавки + разгрузка + праздничные/пики`.
- `Неделя = WEEKNUM(Дата начала смены; 2)`.
- `Месяц = DATE(YEAR(Дата начала смены); MONTH(Дата начала смены); 1)`.
- `Год = YEAR(Месяц)`.

### Поведение job

- `courier_payroll_daily_export`
- Расписание: каждый день `00:30 Europe/Moscow`.
- Период: предыдущий календарный день по Мск.
- Режимы:
  - `dry_run`: получить отчет, нормализовать строки, показать append/upsert
    план без записи;
  - `run`: записать только после включения job.
- Idempotency key:
  - `report_date + pizzeria + employee_id/employee_uuid + shift_start +
    shift_end + source_row_hash`.
- Не оставлять пустые строки между днями.
- При повторном запуске обновлять только разрешенные source/formula поля,
  не затирать ручные правки.

### Критерии приемки

- Dry-run за выбранную дату показывает количество строк по каждой пиццерии,
  отсутствующие пиццерии и дубли.
- Повторный dry-run после run показывает `0` новых строк.
- Формулы в новых строках совпадают с формулами текущей строки-шаблона.
- Ручные поля не перезаписываются.
- В audit log есть source snapshot hash и диапазон записи.

## Функция 2. Реестр выплат курьерам

### Цель

Каждый понедельник в 06:00 Мск сформировать строки за прошлую неделю на листе
`Реестр выплат` той же таблицы на основании листа `Ежедневная выгрузка`.

### Источник

- Основной источник: лист `Ежедневная выгрузка`.
- Период: предыдущая неделя по `Неделя` и датам отчетного периода.
- Внешний Dodo источник для этой job не нужен, если ежедневная выгрузка уже
  заполнена.

### Целевой лист и показатели

Лист `Реестр выплат`, активная структура `A:P`:

- заполняемые поля: `Неделя`, `Id сотрудника`, `Фамилия`, `Имя`,
  `Тип оформления`, `ИНН`, `SUM of Итого зп`;
- формульные поля: `Налог 6% СМЗ 7% ИП`,
  `Сумма с учетом компенсации налога`, `Сумма для перевода ДС`,
  `Кол-во заказов`, `Сальдо конечное`;
- ручные поля, которые нельзя перезаписывать: `Стоимость заказа`,
  `Сальдо начальное`, `Корректировка`, `Дата выплаты`.

### Правила расчетов, обнаруженные в листе

- `Налог = Сумма с компенсацией налога - SUM of Итого зп`.
- Для `Самозанятый`: `Сумма с компенсацией налога = SUM / (100% - 6%)`.
- Для `ГПХ` с ИНН: `Сумма с компенсацией налога = SUM / (100% - 7%)`.
- `Сумма для перевода ДС = ROUNDDOWN(Сумма с компенсацией / Стоимость заказа) *
  Стоимость заказа`.
- `Кол-во заказов = Сумма для перевода ДС / Стоимость заказа`.
- `Сальдо конечное = Сумма с компенсацией - округленная сумма к переводу`.

### Поведение job

- `courier_payroll_weekly_register`
- Расписание: каждый понедельник `06:00 Europe/Moscow`.
- Группировка: `Неделя + Id сотрудника + Тип оформления + ИНН`.
- Записывать только новые уникальные строки или обновлять сумму в уже созданной
  строке, если это тот же сотрудник/неделя и ручные поля пустые либо разрешены.
- Формулы протягивать из строки 2 или из последней корректной строки.
- Не изменять `Стоимость заказа` и `Корректировка`.

### Критерии приемки

- Dry-run показывает список сотрудников, суммы и целевые строки.
- При наличии ручной корректировки job показывает конфликт и не затирает строку.
- Повторный запуск не создает дублей.
- Итоги по сотрудникам совпадают с суммой `Итого зп` в ежедневной выгрузке.

## Функция 3. Анализ затрат ДОДО 2026

### Цель

Заполнить лист `Общий анализ_2026` в таблице
`Анализ затрат ДОДО 2025-2026 dodotm` по аналогии с `Общий анализ_2025`, но
используя только источники 2026 года.

### Источники

Основной источник для этой задачи сейчас не Dodo API, а сама модель Google
Sheets:

- эталон логики: `Общий анализ_2025`;
- целевой лист: `Общий анализ_2026`;
- источники 2026:
  - `Чита 1,2,3_2026`;
  - `Б-1,2,3,4 Бел-ск_2026`;
  - `А-1, А-2, А-3_2026`;
  - `Т-1, Т-2, Т-3_2026`;
  - `Сев-ск_2026`;
  - `Свод по пиццериям_1 кв.2026`;
  - `Свод по пиццериям_2 кв.2026`;
  - будущие `Свод по пиццериям_3 кв.2026` и
    `Свод по пиццериям_4 кв.2026`, если будут созданы.

Dodo API может понадобиться позднее для заполнения региональных source-листов,
но текущий запрос "Общий анализ_2026 по аналогии с 2025" сначала решается
генерацией и проверкой формул.

### Целевые показатели

Лист `Общий анализ_2026`, структура `A:AO`:

- измерения: `Пиццерия`, `Статьи`, `Норма`;
- месячные суммы и проценты: январь-декабрь;
- квартальные итоги и проценты;
- `ИТОГО 2026`;
- `Среднемес. пок-ль, руб.`;
- `Среднемес. пок-ль, %`;
- `Проверка, руб`;
- `Проверка, %`.

В первых строках блока `Архангельск-1` найдены статьи:
`Кол-во заказов по Доставке`, `ВЫРУЧКА ОТ РЕАЛИЗАЦИИ ИС`,
`Касса доставка`, `Касса 1 зал`, `Касса 2 зал`, `Касса 3 зал`,
`Оплата на сайте`, `Доставка`, `Расхождения налички`,
`Комиссия банков за эквайринг`, `ТЕКУЩИЕ РАСХОДЫ`,
`СЕБЕСТОИМОСТЬ (UC)`, `ФУДКОСТ`, `УПАКОВКА`, `ПОТЕРИ`, `ИЗБЫТОК`,
`РАСХОДНИКИ`, `ОПЛАТА ТРУДА (LC)`,
`Заработная плата производства`, `Заработная плата курьеров`,
`Заработная плата управления`, `Налоги и начисления на ФОТ`,
`Деливери кост`, `Стоимость доставки 1 заказа`, `Налоги прочие`,
`Расходы на персонал` и далее полный P&L список.

### Обнаруженная логика формул

На `Общий анализ_2025` строки с данными ссылаются на региональные листы 2025,
например:

- `='А-1, А-2, А-3_2025'!B2`;
- квартал: `=D2+F2+H2`;
- год: `=AH2+Z2+R2+J2`;
- среднемесячный показатель: `=AVERAGE(D2;F2;H2;L2;N2;P2;T2;V2;X2;AB2;AD2;AF2)`;
- проверки: `=AL2*12-AJ2`, `=AM2-AK2`.

На `Общий анализ_2026` сейчас видна структура строк, но в первых строках
формулы еще не заполнены. В региональном листе `А-1, А-2, А-3_2026` данные и
формулы за 2026 уже есть, часть июня пока пустая.

### Поведение job

- `cost_analysis_2026_formula_mapper`
- Запуск: ручной или по расписанию после обновления source-листов.
- Этап 1, read-only:
  - построить карту `пиццерия -> source sheet`;
  - сопоставить строки `Общий анализ_2025` и `Общий анализ_2026`;
  - заменить ссылки `_2025` на `_2026` только там, где найден соответствующий
    source sheet и строка;
  - выявить отсутствующие квартальные листы/месяцы.
- Этап 2, dry-run:
  - показать формулы, которые будут вставлены;
  - показать строки без источника;
  - показать расхождения между списком статей 2025 и 2026.
- Этап 3, run:
  - записать формулы только в пустые/разрешенные ячейки `Общий анализ_2026`;
  - не менять source-листы и ручные значения без отдельного ТЗ.

### Критерии приемки

- Dry-run строит полную карту формул для `Общий анализ_2026`.
- Для каждой строки есть статус: `mapped`, `missing_source`, `manual_required`,
  `formula_conflict`.
- Запись не затрагивает листы-источники.
- После записи проверки `Проверка, руб` и `Проверка, %` либо равны нулю/пусто,
  либо явно попадают в отчет конфликтов.

## Функция 4. Трекер изменений замеров

### Цель

Ежедневно проверять изменения замеров в Dodo IS Office Manager и вести историю
на листе `Замеры_агент` таблицы `Анализ замеров (робот)`.

### Источник

- Основной источник: Dodo IS Office Manager.
- Путь: `Учет -> Замеры -> выбор подразделения`.
- Дополнительное действие в UI: `Действие -> Изменить` в типе сырья.
- Справочник для нормализации: лист `Справочник ID`.

### Целевой лист и показатели

Лист `Замеры_агент`, структура совпадает с `Замеры`, диапазон `A:N`:

- `Ссылка`;
- `Дата ФЗ`;
- `Дата ЛЗ`;
- `Дата последнего изменения`;
- `Актуальность`;
- `Город`;
- `ID сырья`;
- `новый ID`;
- `Тип сырья`;
- `Ед.изм`;
- `Производственно мерная ед.`;
- `Федеральные`;
- `Локальные из ДОДО ИС`;
- `Локальные (формула)`.

На листе `Замеры` формула локального итогового значения:
`=IF(INDIRECT("RC13";0)=0;INDIRECT("RC12";0);INDIRECT("RC13";0))`.

### Правила изменений

- Если изменился федеральный замер:
  - старую актуальную строку сделать `Неактуально`;
  - добавить новую строку `Актуально`;
  - проставить новый федеральный замер и дату ФЗ;
  - локальный замер перенести, если он не изменился.
- Если изменился локальный замер:
  - старую актуальную строку сделать `Неактуально`;
  - добавить новую строку `Актуально`;
  - проставить новый локальный замер и дату ЛЗ;
  - федеральный замер и дату перенести из старой строки.
- Если изменились оба замера:
  - старую строку сделать `Неактуально`;
  - добавить одну новую строку с двумя новыми значениями и датами.
- Если позиция была в листе, но больше не найдена в Dodo IS:
  - считать позицию выведенной;
  - старую строку сделать `Неактуально`.
- Если позиция есть в Dodo IS, но нет в листе:
  - добавить новую строку `Актуально`.

### Ключ сопоставления

Предлагаемый ключ до проверки на реальных данных:

`Город + ID сырья/новый ID + Производственно мерная ед. + Ед.изм`.

Если в Dodo IS стабильным ключом является UUID из ссылки
`/InventoryManager/MaterialMeasurement/edit/<id>`, его нужно хранить и
использовать как главный ключ.

### Поведение job

- `measurements_change_tracker`
- Расписание: ежедневно, точное время согласовать.
- Режимы:
  - read-only snapshot из Dodo IS;
  - dry-run diff against active rows;
  - run: поменять `Актуальность` старых строк и append новых.
- Запись должна быть атомарной: изменение старой строки и добавление новой
  выполняются в одном batch update.

### Критерии приемки

- Dry-run показывает списки `new`, `changed_federal`, `changed_local`,
  `changed_both`, `removed`, `unchanged`.
- Повторный запуск после записи не создает новых строк.
- Старые строки получают только значение `Неактуально`, остальные поля старой
  строки не меняются.
- Новые строки содержат формулу в колонке `N`.
- Есть audit log с количеством изменений по каждому городу.

## Предлагаемые internal API/CLI

Эти функции не включать в public OpenAPI для ChatGPT Action.

- `GET /automation/jobs`
- `POST /automation/jobs/{job_name}/dry-run`
- `POST /automation/jobs/{job_name}/run`
- `GET /automation/jobs/{job_name}/runs/{run_id}`
- `GET /automation/jobs/{job_name}/diff/{run_id}`

Для production лучше закрыть эти endpoints отдельным admin key или вообще
оставить только CLI/systemd timer.

## Приоритет реализации

1. `courier_payroll_daily_export` read-only extractor.
2. `courier_payroll_daily_export` dry-run diff.
3. Google Sheets writer с allowlist и idempotency для `Ежедневная выгрузка`.
4. `courier_payroll_weekly_register` dry-run и writer.
5. `measurements_change_tracker` extractor и dry-run.
6. `measurements_change_tracker` writer.
7. `cost_analysis_2026_formula_mapper` read-only mapping.
8. `cost_analysis_2026_formula_mapper` writer после ручного просмотра diff.
